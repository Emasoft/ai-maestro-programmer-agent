## Problem (re-scoped)

AMPA receives task assignments via AMP messages only. It cannot read its assigned tasks from the kanban, verify task details, or reflect progress (start / submit-for-review / blocked / done) back to the board. Issue #7 specified this as direct `/api/` calls — now forbidden by **R23**. The capability is still wanted; the **transport** must be the frozen CLI layer.

## Required changes — frozen-verb map (replaces every /api/ snippet in #7)

| #7 requirement (was direct `/api/`) | R23-compliant frozen verb | Flags |
|---|---|---|
| Read "my tasks" (`GET …/tasks?assignee=`) | `amp-kanban-list` | `--assignee {agentId}` (already used in Step 1.4); confirm status-filter flag against the live verb |
| Start work (`PUT status=in-progress`) | `amp-kanban-move` | move target task → `in_progress` column; flags TBD against live verb |
| Submit for review + PR (`PUT status=ai-review, prUrl`) | `amp-kanban-move` + `amp-submit-pr` | move → `ai_review`; `amp-submit-pr <repo-path> <title> [--body] [--base]` for the PR |
| Blocked (`PUT status=blocked`) | `amp-task-blocked` | flags TBD against live verb |
| Completed | `amp-task-done` | flags TBD against live verb |

> Exact per-verb flags are deliberately left "TBD against the live verb" — the build MUST invoke each verb (with a real `--id`) and read its `--help`/behavior, not guess. No mocked kanban (per the no-mocked-tests rule).

## Skill wiring (where each transition lands)

1. **Read-back** — `skills/ampa-task-execution/references/op-receive-task-assignment.md` Step 1.4. **DONE in v1.4.0** (`amp-kanban-list --assignee`). No change.
2. **Start → in_progress** — `skills/ampa-task-execution/references/op-implement-code.md` (or the start-of-work op): on beginning work, `amp-kanban-move` the task to `in_progress`. Optional + non-fatal if standalone/unavailable.
3. **Submit for review** — `skills/ampa-orchestrator-communication/references/op-notify-completion.md` + `skills/ampa-github-operations/references/op-create-pull-request.md`: after the AMOA green-light gate, create the PR via `amp-submit-pr`, then `amp-kanban-move` the task to `ai_review` recording the PR URL.
4. **Blocked** — `skills/ampa-orchestrator-communication/references/op-report-blocker.md`: when reporting a blocker, also `amp-task-blocked` the task. (The R6.6/R37.1 escalation prose fixed in v1.4.0 stays as-is.)
5. **Done** — wherever terminal completion is recorded: `amp-task-done`.

## Standalone-mode guard (mandatory)

Every kanban call is **optional** and must degrade silently: if AI Maestro is not running, the programmer has no AID, or the verb is unavailable, skip the call and proceed — AMP messages (orchestrated) or the local operator (standalone) remain authoritative. This mirrors the existing Step 1.4 Note. Never block task execution on a kanban call.
